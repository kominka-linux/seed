# Docs TODO

This file exists to let another coding agent pick up the applet documentation rollout without rediscovering the current design.

## Current Design

- Runtime binaries stay lean.
- Full applet docs are not compiled into normal `seed` or standalone applet binaries.
- Full docs are compiled only behind the non-default `docgen` feature.
- Short help is available at runtime via `--help`.
- Full manpages are generated by `seed-docgen` into `docs/man/*.1`.

Current authored examples:
- [`src/wget_doc.rs`](src/wget_doc.rs)
- [`src/tar_doc.rs`](src/tar_doc.rs)

Shared renderer/types:
- [`src/common/doc.rs`](src/common/doc.rs)

Runtime short-help path:
- [`src/lib.rs`](src/lib.rs)
- [`src/common/help.rs`](src/common/help.rs)

Doc generator:
- [`src/bin/seed-docgen.rs`](src/bin/seed-docgen.rs)

## Quality Bar

Generic fallback manpages are only scaffolding. They are not real documentation.

For an applet to count as documented, its authored page should include:
- A real one-line summary.
- A real synopsis that matches supported invocation forms.
- A concise description of what the applet actually does.
- An option table covering the flags users are likely to need in scripts.
- Two to four examples of real usage.
- `SEE ALSO` entries when there are obvious related tools.

Do not add filler prose.
Do not restate obvious flags in vague language.
Do not add low-value tests like “help exits success” or “short help mentions man page”.

## How To Add Real Docs For One Applet

Use `wget` and `tar` as the model.

### 1. Add a sidecar doc module

Create `src/<applet>_doc.rs`.

Example shape:

```rust
use crate::common::doc::{AppletDoc, DocExample, DocOption};

pub fn doc() -> &'static AppletDoc {
    &DOC
}

const DOC: AppletDoc = AppletDoc {
    name: "grep",
    section: "1",
    summary: "search files for matching lines",
    usage: &[
        "grep [OPTIONS] PATTERN [FILE...]",
    ],
    description: &[
        "grep searches each input file for lines matching PATTERN.",
    ],
    options: &[
        DocOption {
            flags: "-n",
            help: "Prefix each matching line with its line number.",
        },
    ],
    examples: &[
        DocExample {
            command: "grep -n main src/lib.rs",
            help: "Show matching lines with line numbers.",
        },
    ],
    see_also: &["sed(1)", "find(1)"],
};
```

### 2. Export it from `src/lib.rs`

Add:

```rust
#[cfg(feature = "docgen")]
pub mod grep_doc;
```

Keep this behind `docgen`.

### 3. Teach `seed-docgen` about it

Update [`src/bin/seed-docgen.rs`](src/bin/seed-docgen.rs) so `man <applet>` uses the authored page instead of the generic fallback.

Add a match arm like:

```rust
"grep" => seed::common::doc::render_manpage(
    seed::grep_doc::doc(),
    concat!("seed ", env!("CARGO_PKG_VERSION")),
    "",
),
```

If there is no authored page yet, the applet should continue to fall back to `render_generic_manpage(...)`.

### 4. Generate the manpage

Run:

```sh
make manpages
```

That updates `docs/man/<applet>.1`.

### 5. Add one focused test

Add one meaningful unit test in the sidecar doc module.

Good examples:
- Assert the doc name is correct.
- Assert an important option entry exists.
- Assert a newly-added compatibility flag is documented.

Bad examples:
- Help exits successfully.
- The short help mentions `man`.
- The rendered manpage contains basic section headings already covered by shared tests.

## Writing Guidance

When writing docs for an applet:

- Read the applet implementation first.
- Read the applet's tests, especially shell tests under `tests/busybox/<applet>/`.
- Document what the applet actually supports now, not what GNU or BusyBox supports in general.
- Mention compatibility quirks only when they matter in practice.
- Prefer examples that reflect real build-script or shell usage.
- Keep option help specific and behavior-oriented.

For synopsis lines:
- Include the common invocation forms people will actually use.
- Include separate lines if the applet has materially different modes.
- Avoid pseudo-syntax that hides required operands.

For examples:
- Favor script-visible behavior.
- Show one normal case and one compatibility-sensitive case if relevant.
- Keep examples short enough to read in a manpage.

## Short Help

Runtime `--help` is intentionally short.

- Generic applets use the central fallback in [`src/common/help.rs`](src/common/help.rs).
- Special applets may provide a custom short help string when the generic fallback is too weak.

Do not duplicate full manpage content in runtime help.

## Validation

For one applet doc change, the normal checks are:

```sh
cargo fmt --all --check
docker compose run --rm dev cargo test --quiet
cargo run --quiet --features docgen --bin seed-docgen -- man <applet> | sed -n '1,80p'
```

If you update multiple manpages, also run:

```sh
make manpages
```

## Commit Discipline

- One applet per commit.
- Keep doc-only changes separate from behavior changes when practical.
- Regenerated `docs/man/<applet>.1` should land in the same commit as the corresponding `src/<applet>_doc.rs`.
